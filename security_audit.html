▶ High LLM-Generated Bash Commands Executed on Host bash.rs / bash_validation.rs

Description

The tool accepts bash commands generated by the LLM and executes them via std::process::Command. While heavily validated, the LLM could be influenced by prompt injection to craft commands that bypass heuristic checks.

Affected Code

// bash_validation.rs:103 — Read-only validation gate
pub fn validate_read_only(command: &str, mode: PermissionMode) -> ValidationResult {
    if mode != PermissionMode::ReadOnly {
        return ValidationResult::Allow;
    }
    let first_command = extract_first_command(command);
    for &write_cmd in WRITE_COMMANDS { ... }
    for &state_cmd in STATE_MODIFYING_COMMANDS { ... }
}

Current Guardrails

Read-only command blocklist (30+ commands)

Destructive pattern detection (fork bombs, rm -rf /)

Permission mode gating per command

Path traversal heuristics

Sandbox support (Linux only)

Gap Analysis

Blocklist approach is inherently incomplete — novel command combinations, aliased binaries, or encoded payloads (e.g., python3 -c "...", perl -e "...") can bypass first-command extraction. The extract_first_command parser splits on whitespace which may miss shell metacharacters.

Recommendation: (1) Complement blocklist with allowlist of known-safe commands for read-only mode. (2) Add sandbox enforcement as default on Linux. (3) Consider integrating seccomp-bpf filters for fine-grained syscall control. (4) Log all executed commands to an audit trail for post-hoc review.
▶ High MCP Server Subprocess Inherits Unsanitized Environment mcp_stdio.rs / mcp_client.rs

Description

MCP servers are spawned as child processes via tokio::process::Command. Environment variables from configuration are passed directly without sanitization. A malicious .claw.json project config could inject environment variables like LD_PRELOAD or PATH to hijack execution.

Affected Code

// mcp_client.rs — MCP transport config
pub struct McpStdioTransport {
    pub command: String,
    pub args: Vec<String>,
    pub env: BTreeMap<String, String>, // passed to subprocess directly
}

Current Guardrails

Environment scoped to config-specified keys only

No blocklist for dangerous env vars (LD_PRELOAD, PATH, etc.)

Project-level config can define MCP servers (untrusted source)

Recommendation: (1) Blocklist dangerous environment variable names (LD_PRELOAD, LD_LIBRARY_PATH, DYLD_INSERT_LIBRARIES, PATH). (2) Require user confirmation before spawning MCP servers defined in project-level config. (3) Run MCP subprocesses in a restricted environment rather than inheriting the full parent env.
▶ Medium Sandbox Namespace Isolation Unavailable on macOS sandbox.rs

Description

Linux namespace sandboxing (unshare) provides process/network isolation, but macOS has no equivalent. On macOS, the sandbox is effectively a no-op for namespace and network isolation, leaving only filesystem boundary checks as protection.

Affected Code

// sandbox.rs:164
let namespace_supported = cfg!(target_os = "linux") && unshare_user_namespace_works();
let network_supported = namespace_supported; // false on macOS

Current Guardrails

Filesystem boundary enforcement (all platforms)

Fallback reason reporting when capabilities unavailable

No macOS sandbox-exec or App Sandbox integration

Recommendation: Investigate macOS sandbox-exec (seatbelt) profiles or the App Sandbox entitlement API to provide filesystem and network isolation on macOS. At minimum, document the reduced security posture for macOS users.
▶ Medium OAuth Tokens Persisted to Disk Without Encryption oauth.rs

Description

OAuth access tokens and refresh tokens are serialized to disk as plaintext JSON. Any process or user with filesystem read access can extract these credentials.

Affected Code

// oauth.rs:13 — Token struct persisted via serde
pub struct OAuthTokenSet {
    pub access_token: String,
    pub refresh_token: Option<String>,
    pub expires_at: Option<u64>,
    pub scopes: Vec<String>,
}

Current Guardrails

PKCE (S256) for authorization code exchange

Token masking in log output

OS file permissions (relies on user-level access control)

No encryption-at-rest or OS keychain integration

Recommendation: Integrate with platform keychains (macOS Keychain, Linux Secret Service / libsecret, Windows Credential Manager) for token storage. Alternatively, encrypt the token file with a key derived from machine-specific entropy.
▶ Medium Bash Path Validation Uses Heuristic String Matching bash_validation.rs:360

Description

The validate_paths function detects ../ traversal and home-directory references using simple string matching. This can be bypassed with encoded paths, shell variable expansion ($(...)), or symlink-relative references.

Affected Code

// bash_validation.rs:360
pub fn validate_paths(command: &str, workspace: &Path) -> ValidationResult {
    if command.contains("../") {
        let workspace_str = workspace.to_string_lossy();
        if !command.contains(&*workspace_str) {
            return ValidationResult::Warn { ... };
        }
    }
    if command.contains("~/") || command.contains("$HOME") {
        return ValidationResult::Warn { ... };
    }
}

Gap

Does not detect: $(cd /etc && cat passwd), eval-based expansion, backtick substitution, or symlink chains that reach outside workspace.

Recommendation: Expand validation to cover shell substitution patterns ($(), backticks, eval, source). For write commands, resolve the actual target path after shell expansion rather than pattern-matching the raw command string.
▶ Medium Hooks Can Override Permission Decisions hooks.rs / permissions.rs

Description

Hooks execute shell commands and can return PermissionOverride::Allow, which bypasses normal permission evaluation. While hooks are user-configured, a compromised or malicious project-level config could define hooks that auto-approve dangerous operations.

Affected Code

// permissions.rs:196 — Hook overrides evaluated early
match context.override_decision() {
    Some(PermissionOverride::Allow) => {
        if allow_rule.is_some() || current_mode == PermissionMode::Allow
            || current_mode >= required_mode {
            return PermissionOutcome::Allow;
        }
    }
}

Mitigation Note

The code does check ask_rule even when a hook says Allow (line 220), which limits the override scope. Deny rules always take precedence (line 182).

Recommendation: (1) Clearly separate user-level hooks from project-level hooks in trust evaluation. (2) Add an audit log entry whenever a hook overrides a permission check. (3) Consider requiring user confirmation when project-level hooks attempt Allow overrides.
▶ Low No Rate Limiting on Tool Execution runtime (general)

Description

There is no built-in rate limiting on how many tool calls the LLM can make per unit of time. A misbehaving model loop could generate rapid file operations or bash commands.

Current Mitigations

Timeout support per individual bash command

File size limits (10 MB read/write)

Recommendation: Add configurable per-minute limits on tool invocations, particularly for bash execution and file write operations.
▶ Low Write Redirection Detection Is Simple String Match bash_validation.rs:97

Description

Write redirections (>, >>, >&) are detected by checking if the command string contains these characters. This triggers false positives for quoted strings and misses redirections inside subshells or variables.

Affected Code

Recommendation: Use a shell-aware tokenizer to distinguish actual redirections from characters inside quoted strings or variables.

▶ Info Session Token Stored at Predictable Path remote.rs:8

Description

The default session token path is /run/ccr/session_token. This is a well-known location that any process on the machine could read, though /run typically has restricted permissions.

▶ Info Proxy Environment Allowlist Is Broad remote.rs:11

Description

The UPSTREAM_PROXY_ENV_KEYS allowlist includes TLS certificate bundle paths (SSL_CERT_FILE, NODE_EXTRA_CA_CERTS, etc.). A malicious proxy config pointing these at a crafted CA bundle could enable MITM interception of API traffic.

Recommendation: If not required for enterprise proxy scenarios, consider requiring explicit opt-in for custom CA bundle paths. Log when custom CA bundles are in use.

Notable Security Strengths

Strength Zero Unsafe Code — Workspace-Wide Forbid Cargo.toml:15

The workspace root sets unsafe_code = "forbid", preventing any crate in the workspace from using unsafe blocks. This eliminates entire classes of memory safety vulnerabilities (buffer overflows, use-after-free, data races).

// Cargo.toml
[workspace.lints.rust]
unsafe_code = "forbid"
Strength Defense-in-Depth: 5-Layer Security Architecture

Security controls are layered so that no single failure is catastrophic: input validation → permission system → hooks → filesystem boundary → sandbox. Each layer independently prevents different attack classes.

Strength Comprehensive Path Traversal Protection with Symlink Detection file_ops.rs:28-43, 604-615

Path canonicalization via canonicalize() resolves all symlinks and relative components before boundary checking. A dedicated is_symlink_escape() function catches symlinks that resolve outside the workspace. Both are covered by unit tests.

Strength 1004-Line Bash Validation Pipeline bash_validation.rs

The validation pipeline mirrors upstream TypeScript with 6 sub-validators: read-only enforcement, destructive command warning, mode validation, sed validation, path validation, and semantic command classification with 8 intent categories.

Strength rustls TLS — No OpenSSL Dependency api/Cargo.toml

Using rustls instead of OpenSSL eliminates a large class of C-based TLS vulnerabilities. The entire TLS stack is pure Rust with memory-safe guarantees.

Strength OAuth PKCE with CSRF State Parameter oauth.rs

OAuth implementation uses PKCE with S256 challenge method and includes a state parameter for CSRF protection. This follows current best practices for public client OAuth flows (RFC 7636).

Strength Deny Rules Always Take Precedence permissions.rs:182

The permission system evaluates deny rules first, before any allow rule or hook override. This fail-safe design ensures that explicitly blocked operations cannot be overridden by any other mechanism.

Strength Strict Clippy Lints (pedantic + all = warn) Cargo.toml:17-19

Workspace-wide Clippy configuration with both all and pedantic lint groups set to warn catches subtle code quality issues that could become security problems (unchecked casts, implicit conversions, etc.).
